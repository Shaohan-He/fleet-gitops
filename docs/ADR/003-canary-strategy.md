# ADR-003: 金丝雀发布策略设计

- **状态：** 已采纳
- **日期：** 2026-05-30
- **决策者：** 何少涵

---

## 背景

需要设计金丝雀发布的步骤数量、权重分布、Analysis 指标选择。

## 决策

**采用 20% → 50% → 100% 三阶段金丝雀，配合 4 项 Analysis 指标。**

## 细节

### 为什么是三阶段而非两阶段？

| 策略 | 步骤 | 风险 |
|------|------|------|
| 两阶段 | 10% → 100% | 10% 可能没收到真实流量，样本太小 |
| 两阶段 | 50% → 100% | 50% 爆炸半径过大 |
| 三阶段 | 20% → 50% → 100% | ✅ 兼顾安全性与样本量 |

三阶段的优势在于：20% 控制爆炸半径（影响面可控），50% 作为二次验证（扩大样本排除偶然性），多一层安全网。

### 为什么 Analysis 间隔是 120 秒？

- 太短（30s）：指标还没稳定，容易误报
- 太长（5min）：发布太慢
- 120s：Prometheus 抓取间隔 30s × 4 次 = 足够判断趋势

### 为什么 failureLimit = 2 而不是 1？

- 防止 Prometheus 短暂不可用导致误回滚
- 需要连续两次确认才判定失败
- 2 次 × 30s 间隔 = 给了系统 60s 自愈时间

### 为什么 4 项指标？

四项指标覆盖三个层级：
1. 基础设施层：NHW 节点健康分
2. 应用层：分配成功率 + 分配延迟 P99
3. 控制层：熔断器状态

三层交叉验证避免单层指标误判。

### 为什么不用 autoPromotion（自动放行）？

`autoPromotionEnabled: false` 意味着每个 Analysis 通过后需要手动确认才进入下一步。这遵循 SRE 最佳实践——自动化应该建议行动而不是替代决策，保留运维人员的最终判断权。

## 后果

### 正面
- 安全问题：任意 Analysis 失败自动回滚
- 可观测：每个步骤的状态在 ArgoCD UI 可见
- 可审计：每次金丝雀的结果都记录在 Rollout 的 status 中

### 负面
- 发布速度慢（~6 分钟的金丝雀窗口）
- 需要 Prometheus + NHW 两个外部依赖

### 风险缓解
- 如果 NHW 不可用，AnalysisTemplate 中的 web metric 会 fallback 到 failureLimit
- Prometheus 用 recording rules 预计算指标，减少 Analysis 查询延迟
